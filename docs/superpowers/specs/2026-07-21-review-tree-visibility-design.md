# 복습 나무 현황 가시화 — 브레인스토밍 결정사항 (설계 인계용)

> 2026-07-21 브레인스토밍 세션 산출물. 상세 설계는 후속 세션(opus)에서 진행.

## 배경 / 문제

복습(간격 반복) 시스템의 메타포는 "오답에 물 주기 → 나무 키우기"인데,
사용자에게 보이는 건 ①오늘 복습 목록(`GET /api/reviews/today`) ②졸업 총계(`graduatedTrees` 카운터)뿐.
**자라는 중인 나무들의 현황이 어디에도 안 보여** 성취의 중간 과정이 비어 있다.

제작자(사용자) 요구:
1. **자라는 현황 목록** — 어떤 문제가 몇 단계인지 (대시보드)
2. **졸업 연출** — 나무가 되는 순간의 성취감
3. **문제 상세 연동** — 복습 풀이 화면에서 "이 문제 물 몇 번 줬는지" 표시
4. **오답노트 연동** — 오답노트 페이지에서 문제별 물 준 횟수/나무 여부 표시
5. "어떤 문제가 나무가 됐고 물 몇 번 줬는지"까지 보여주는 건 **과하지 않다**로 합의 — 메타포의 완성

## 확정된 설계 결정

- **물 준 횟수 = C안 (둘 다 저장)**: `총 복습 시도 수`와 `맞힌 복습 수`를 모두 저장. 표시는 프론트 선택.
  (오답노트 상세에서 "물 7번, 그중 3번 흡수" 같은 표현 가능)

## 현재 구조 (설계 시 참고)

- `ReviewItem` (`review/domain/ReviewItem.java`): user_id+quiz_id UK, stage(0~2), due_date.
  간격 {3,7,14}일. **졸업 시 row 삭제** + users 테이블 카운터(graduatedTrees) 적립.
- 복습 채점: `POST /api/reviews/{quizId}/answer` (`ReviewService.answerReview`).
  ⚠️ REQUIRES_NEW/FK 락 순서 주석 있음 — 수정 시 락 순서 불변 유지 필수.
- 잔디 API `GrassResponse` 가 graduatedTrees 노출 중. 스트릭과 잔디는 의도적으로 다른 축.

## 설계 세션에서 풀어야 할 것 (미결)

1. **졸업 보존 방식**: row 삭제 대신 보존 필요 ("나무 목록"의 전제).
   후보: (a) review_item 에 graduated_at 컬럼 추가하고 삭제 안 함 (b) 별도 graduated 테이블.
   기존 쿼리들(uk, due 조회)이 졸업 row 를 제외하도록 조건 추가 필요 여부 포함.
2. **물 카운터 2종 컬럼** 추가 위치·마이그레이션(prepare-server.sh 멱등 가드 패턴 준수).
3. **API 형태**: 현황 목록(신규 엔드포인트 vs GrassResponse 확장), 복습 풀이 응답에 물 이력 포함,
   오답노트(attempt 목록 API)에 review 상태 join 방식.
4. **졸업 연출**: 백엔드는 복습 채점 응답의 graduated 정보 확장으로 충분한지 (연출 자체는 프론트).
5. users.graduatedTrees 카운터와 보존 row 의 정합 (이중 진실 방지 — 카운터 폐기하고 COUNT 로 대체할지).
